# DTAOS Knowledge Specification

> Define how knowledge is stored, maintained, and reused within DTAOS.

---

# Philosophy

DTAOS does not accumulate documents.

DTAOS accumulates reusable knowledge.

Every piece of knowledge must have a single purpose and a single owner.

One Knowledge, One Location.

---

# Knowledge Architecture

Project

↓

Case

↓

Retrospective

↓

Lesson

↓

Workflow

↓

Technology

↓

Template

↓

Next Project

Knowledge evolves through projects.

Projects do not own knowledge.
Projects validate knowledge.

---

# Documentation Types

## Case

### Purpose

Record the execution of a project.

### Contains

- Objectives
- Scope
- Deliverables
- Decisions
- Outputs

### Does NOT Contain

- Detailed Lessons
- Technology Knowledge
- Workflow Definitions

---

## Retrospective

### Purpose

Review the completed project.

### Contains

- What worked well
- What should improve
- Newly validated principles
- Next actions
- Links to Lessons

### Does NOT Contain

- Technology explanation
- Standard operating rules
- Workflow definitions

---

## Lesson

### Purpose

Define reusable operating rules.

### Contains

- Rule
- Why
- Reuse
- Related

### Does NOT Contain

- Project history
- Personal opinion
- Temporary decisions

Lessons describe

NOT

"What happened"

Lessons describe

"What becomes the new standard."

---

## Workflow

### Purpose

Define repeatable execution processes.

### Contains

- Input
- Process
- Output
- Validation

### Does NOT Contain

- Technology
- Project review

---

## Technology

### Purpose

Maintain technical knowledge.

### Contains

- Principles
- Mechanisms
- Validation
- Data
- Applications
- References

### Does NOT Contain

- Project-specific decisions
- Workflow

Technology exists independently of projects.

---

## Template

### Purpose

Store reusable assets.

### Contains

- PPT Master
- Prompt
- HTML Prototype
- Grid
- Typography
- Color
- Components

Templates are implementation assets.

---

# Ownership

Each piece of knowledge has only one owner.

Case owns projects.

Retrospective owns reviews.

Lesson owns operating rules.

Workflow owns execution processes.

Technology owns technical knowledge.

Template owns reusable assets.

No information should exist in multiple locations.

Reference instead of duplicate.

---

# Standard Rule

Every completed project must produce

- Retrospective
- Updated Lessons
- Updated Workflow (if changed)
- Updated Technology (if changed)
- Updated Templates (if changed)

Projects improve DTAOS.

---

# Writing Principle

Knowledge must be

- Reusable
- Verifiable
- Maintainable
- Independent

Avoid project-specific descriptions whenever possible.

Prefer standards over experiences.

---

# Golden Rule

One Knowledge.

One Location.

One Owner.
