# DTAOS Lessons

## Purpose

Lessons store validated operating rules.

A Lesson is created only after a rule has been verified through a real project.

Lessons are not project histories.

Lessons define standards that can be reused in future projects.

---

## Philosophy

Projects validate knowledge.

Lessons preserve knowledge.

DTAOS evolves by accumulating reusable standards, not project memories.

---

## Workflow

Project

↓

Review

↓

Lesson

↓

Git Commit

↓

Boot Recovery

---

## Lesson Rules

A Lesson must satisfy all of the following.

- Verified through a real project
- Reusable
- Independent from a single project
- Maintainable
- Referenced instead of duplicated

If a finding cannot be reused,
it is not a Lesson.

---

## Writing Principle

Write standards.

Do not write project histories.

Good

Prototype before Deliverable.

Bad

In Project P001, HTML was used first.

---

## Quality Standard

A Lesson should be understandable
without reading the original project.

A Lesson should prevent the same mistake
from happening again.

---

## Goal

Every completed project should improve DTAOS.

Every Lesson should become part of the DTAOS operating system.

---

## Golden Rule

A Lesson is written for the next project,

not for the previous one.
